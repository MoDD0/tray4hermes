# Tray4Hermes agent instructions

## Completed-work commits and versions

Use Semantic Versioning with Conventional Commits for every completed work unit:

- `fix:` or `perf:` → PATCH
- `feat:` → MINOR
- `type!:` or a `BREAKING CHANGE:` footer → MAJOR
- `docs:`, `test:`, `chore:`, `refactor:` without a breaking change → no bump

The installed `prepare-commit-msg` hook derives the required bump and rejects
a missing or wrong version. Before committing, run:

```bash
python scripts/versioning.py patch|minor|major
```

Include the version change in the same commit as implementation and tests. Run the full test/lint/security gates before committing. Do not tag, publish, or push unless the user explicitly requests it.

The authoritative policy is in `CONTRIBUTING.md`; the version source of truth is `src/tray4hermes/__init__.py`.

## Documentation rule

Every user-visible behavior change must update the relevant documentation in
the same work unit and commit. Keep `docs/i18n/en.md` canonical, update
`docs/i18n/cs.md`, then run `python scripts/i18n_build.py` so generated README
files stay synchronized. Do not treat documentation as an optional follow-up.

 ## README parity rule (en ↔ cs)

 The two-language README is a single source of truth split into two
 files: `docs/i18n/en.md` (canonical) and `docs/i18n/cs.md` (translation).
 The committed `README.md` and `docs/README.cs.md` are generated by
 `scripts/i18n_build.py` — never edit them by hand.

 Whenever you change one, the other must change in the same commit:

 1. The set of image references in both files must match. The parity
    test `tests/test_readme_parity.py::test_en_and_cs_have_the_same_images`
    fails on the build server if they drift.
 2. Image paths in the source files must be **relative to the repo
    root** (e.g. `docs/images/preview.png`). Never use
    `../images/X.png` or `./X.png` from `docs/i18n/*.md` — the build
    script rewrites paths per compiled target, and a wrong relative
    base lands the asset outside the repo.
 3. The H1 in both files must be `# tray4hermes` (canonical) and
    `# tray4hermes (česky)` (translation). The version placeholder
    (`<!-- tray4hermes:version -->`) follows immediately after the
    title; the build script substitutes the live `__version__`.
 4. The `<!-- i18n:available-languages:START --> … END -->` block is
    auto-managed — don't edit it by hand. The language banner above
    the H1 is rewritten by the build script too.
 5. If you add a new screenshot, drop it in `docs/images/` **and**
    add a tuple to `_IMAGE_LOCATIONS` in `scripts/i18n_build.py` —
    the build then fails fast on a missing asset instead of silently
    shipping a broken README.
 6. Run `pytest tests/test_readme_parity.py` after every README
    edit. If it fails, the source files are out of sync — fix the
    source, not the generated READMEs.
 7. Run `python scripts/i18n_build.py` to regenerate `README.md`
    and `docs/README.cs.md`, and commit the generated files in the
    same commit as the source edit. The version badge inside the
    generated README must always reflect the current
    `src/tray4hermes/__init__.py::__version__`.
